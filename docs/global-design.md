# 产品想法调研工具 PRD 第一版

## 1. 文档信息

- 文档名称：产品想法调研工具 PRD 第一版
- 文档定位：产品需求文档 / 高层产品设计文档
- 产品阶段：0 到 1
- 文档目标：定义一款帮助用户从全网验证产品想法、识别竞品、提炼用户需求与不满、评估进入价值并生成切入策略的产品系统。[cite:9][cite:15][cite:21]

## 2. 产品概述

该产品是一个“产品想法情报与决策系统”，面向创业者、独立开发者、产品经理与研究型团队。用户输入一个产品想法后，系统会在搜索引擎、社交平台、社区、应用商店、软件点评站、电商平台与行业信息源中采集相关证据，再将这些证据结构化为需求、竞品、满意点、不满点、改进空间、风险与机会等维度，最终输出是否值得做、适合如何做、下一步先验证什么的系统建议。[cite:9][cite:10][cite:15][cite:17][cite:20][cite:21][cite:23]

该产品的核心价值不在于“搜索信息”本身，而在于将跨平台、异构、噪声很大的原始信息转化为可决策的统一结论。传统调研动作通常分散在 Google、Reddit、应用商店、G2、Product Hunt、评论区与内容站点中，信息碎片化严重，且研究者容易凭少量样本做过度判断；本产品的任务是降低这一过程中的时间成本、遗漏风险与主观偏差。[cite:20][cite:21][cite:23]

## 3. 产品目标

### 3.1 核心目标

1. 帮助用户快速判断一个产品想法是否存在真实需求。[cite:15][cite:21]
2. 帮助用户识别当前市场中的竞品分布、成熟度与拥挤程度。[cite:9]
3. 帮助用户从评论、帖子与讨论中提炼满意点、不满点与改进点。[cite:9][cite:17][cite:23]
4. 帮助用户判断该想法更适合“直接做、差异化做、换细分做、先验证后做”中的哪一种路径。[cite:13][cite:18][cite:21]
5. 帮助用户生成可执行的下一步验证方案，而不止停留在报告层面。[cite:18][cite:21]

### 3.2 商业目标

1. 成为独立开发者与小团队的前置研究工作台。
2. 提供高频的“想法验证—竞品跟踪—机会发现”工作流。
3. 形成平台级数据资产，包括竞品库、评论主题库、平台信号库与研究项目库。[cite:9][cite:17]

## 4. 目标用户

| 用户类型 | 典型特征 | 核心需求 |
|---|---|---|
| 独立开发者 | 资源有限、需要快速判断值不值得做 | 低成本验证需求、找到切入口 |
| 创业者 | 有多个产品方向待筛选 | 比较不同想法的市场价值与进入难度 |
| 产品经理 | 负责探索新方向或新功能 | 收集需求、竞品、用户反馈并形成方案 |
| 研究团队/咨询团队 | 需要批量做市场与竞品研究 | 提效、结构化输出、沉淀项目知识 |

## 5. 核心问题定义

当前用户在产品想法验证阶段面临以下问题：

- 信息过于分散，必须在多个平台手动搜索与整理。[cite:10][cite:20]
- 很难系统比较“需求强度”“竞品拥挤度”“用户不满密度”等不同维度。[cite:9][cite:21]
- 用户评价和社区讨论噪音很大，难以从中提炼共性结论。[cite:17][cite:23]
- 很多分析只能停留在“有竞品/没竞品”，无法继续回答“那应该怎么做”。[cite:13][cite:18]
- 容易因样本偏差或平台偏差而得出错误判断，缺乏统一的证据权重机制。[cite:20][cite:23]

## 6. 产品定位

一句话定位：一款帮助用户从全网真实信号中验证产品想法、识别竞品空白、评估进入价值并生成切入策略的研究决策工具。[cite:15][cite:21]

产品边界上，它不是通用搜索引擎，也不是单纯的舆情工具，更不是只做应用商店 ASO 的工具。它的核心差异在于：围绕“一个产品想法是否值得做以及如何做”这一目标，统一串联多平台研究、信号提炼、评分评估与策略生成四个过程。[cite:9][cite:13][cite:20][cite:21]

## 7. 使用场景

### 7.1 场景一：验证一个全新想法

用户输入一个尚未明确形态的产品想法，希望知道市场是否存在真实需求，以及用户目前如何解决该问题。系统需要重点识别搜索需求、社区求助、原始替代方案与潜在付费意愿，并判断该领域是“空白机会”还是“伪需求”。[cite:15][cite:18][cite:21]

### 7.2 场景二：进入一个已有竞品的赛道

用户已经知道有若干竞品存在，但不清楚是否仍值得进入。系统需要识别竞品格局、头部集中度、评论中的高频不满点与可差异化切口，帮助用户选择“更便宜、更简单、更垂直、更快、更适合某类人群”的进入方式。[cite:9][cite:13][cite:17][cite:23]

### 7.3 场景三：为已有产品寻找优化方向

用户已有产品，希望借助全网反馈看竞争对手做得好什么、做得差什么，以及用户还在抱怨哪些问题。系统需要输出可参考的功能改进、体验优化与定位升级建议。[cite:9][cite:17][cite:23]

### 7.4 场景四：批量比较多个想法

用户同时有多个方向，希望按统一标准比较其价值、门槛、风险与可验证性。系统需要以统一评分模型输出对比结果，而不是分别给几份难以横向比较的长报告。[cite:13][cite:18][cite:21]

## 8. 产品原则

1. 先证据，后结论：所有结论必须可回溯到具体平台与原始内容。
2. 先结构化，后智能化：先统一信息结构，再做评分和建议生成。
3. 防止过度自信：系统必须显示证据量、样本来源与不确定性提示。[cite:20][cite:23]
4. 结论必须可行动：每次研究输出都要落到“下一步做什么”。[cite:18][cite:21]
5. 平台全面但分类型加权：不同产品类型应使用不同平台优先级。[cite:9][cite:10][cite:17]

## 9. 总体产品架构

产品分为六个核心引擎：

1. 想法建模引擎：把模糊想法转成标准研究对象。[cite:15][cite:21]
2. 全网采集引擎：按平台类型抓取与整合证据。[cite:9][cite:10][cite:20]
3. 信号抽取引擎：从文本中提炼需求、痛点、替代、满意、不满、改进与风险信号。[cite:9][cite:17][cite:23]
4. 评估引擎：按统一维度评估价值与进入难度。[cite:13][cite:18][cite:21]
5. 策略生成引擎：根据不同局面生成切入建议。[cite:13][cite:18]
6. 验证行动引擎：生成访谈、落地页、原型、广告与手工验证等下一步计划。[cite:18][cite:21]

## 10. 数据源体系

### 10.1 一级数据源分类

| 数据源类型 | 作用 | 代表平台 |
|---|---|---|
| 搜索引擎与趋势 | 识别主动搜索需求与问题表达 | Google、Bing、Google Trends [cite:10] |
| 社区与问答 | 识别真实问题、替代方案与长期抱怨 | Reddit、Quora、知乎、贴吧 [cite:20] |
| 社交媒体 | 识别讨论热度、传播点与情绪 | X、TikTok、YouTube、LinkedIn [cite:10][cite:14] |
| 应用商店 | 识别竞品、评分、更新频率与评论 | App Store、Google Play、Chrome Web Store [cite:9] |
| 软件点评站 | 识别 B2B 工具满意度、购买理由与不满点 | G2、Capterra、Trustpilot、AlternativeTo [cite:17][cite:23] |
| 电商与交易平台 | 识别购买意图、差评、价格敏感度 | Amazon、Etsy、淘宝、京东 [cite:10][cite:17] |
| 内容与 SEO 结果 | 识别教育成本、成熟度与内容供给格局 | 博客、媒体站、教程站 [cite:15] |
| 行业与外部环境 | 识别趋势、合规、技术变化与外部风险 | 行业报告、政策公告、技术新闻 [cite:12] |

### 10.2 平台优先级逻辑

系统需根据产品类型自动调整平台权重。例如，B2B SaaS 更强调 G2、Capterra、LinkedIn 与 Reddit；消费类 App 更强调应用商店、TikTok、YouTube 与 Reddit；插件与开发工具则更强调插件市场、GitHub、开发者社区与专业论坛。[cite:9][cite:17][cite:23]

用户输入可以包括：
- 一句话产品想法。
- 可选的目标用户描述。
- 可选的使用场景描述。
- 可选的产品形态说明。
- 可选的已知竞品名称。

| Constraints | 预算、技术、地区、平台等限制 |

## 12. 功能模块

### 12.1 模块一：想法建模

功能目标：将模糊想法变为可研究对象，减少后续搜索偏差。[cite:15][cite:21]

核心功能：
- 想法拆解：用户、场景、任务、替代方案、价值主张。
- 自动识别产品类别，如 SaaS、消费 App、AI 工具、插件、教育产品等。

**四层查询词汇发现架构**（解决词汇鸿沟问题——产品描述词汇与用户搜索词汇之间的范式断裂）：

1. **描述词汇层**（原有）：同义词、近义词、场景词、竞品词。适用于产品有成熟类别名的场景（如 todo app、time tracker）。

2. **痛苦表达层**（新增）：从用户的原始痛苦语言出发生成查询，而非从产品描述出发。用户搜索的是他们的痛苦，不是解决方案的名称。模板示例：
   - "how do I [手工完成产品所做任务的描述]"
   - "is there a way to [job description without naming the tool]"
   - "how to find [what the product discovers] without [expensive/complex existing method]"
   - "[problem description] tool / app"
   - "best way to [job] before [consequence]"

   对于无成熟类别名的产品（如 Aether 自身），痛苦表达层是发现相关内容的首要通道。

3. **替代行为层**（新增）：搜索正在手工完成这件事的用户——他们就是目标用户，他们的帖子就是需求信号，他们提到的工具就是竞品。模板示例：
   - "I manually [手工替代行为描述]"
   - "using [非专业工具] to [任务描述]"
   - "spending time [手工过程] for [目标]"
   - "anyone else [手工行为] to [目标]"

   替代行为层在 Reddit、Quora 等社区平台特别有效。

4. **社区词汇层**（新增）：同一需求在不同平台社区有完全不同的表达词汇，需按平台生成本地化词汇变体。例如：
   - r/startups 使用 "market validation"、"competitive landscape"、"PMF"
   - r/entrepreneur 使用 "competitor analysis"、"market research"
   - IndieHackers 使用 "idea validation"、"find competitors before building"
   - Product Hunt 使用 "alternative to X"、"for [specific user]"
   - App Store 使用功能词、行业词

   系统需为每个平台维护词汇变换规则表，想法建模生成基础词汇后根据平台自动变换词汇风格。

**迭代精化机制**（新增）：对于难以用关键词定位的产品，采用两轮搜索架构：
- 第一轮（宽网捕词）：用原始模糊关键词搜索，重点从结果中提取词汇而非提取信号
- 词汇提取：从第一轮结果中提取高频共现词、与产品核心 job 相关的原生词汇、竞品被提及时的词汇语境
- 第二轮（精准搜索）：使用从真实用户讨论中提取的社区原生词汇重新构造查询，搜索精准度大幅提升

**功能三角定位**（新增）：对于无明确类别名的产品，不直接搜索竞品名，而是搜索竞品被提及的语境：
- "alternatives to [手工方法]"
- "[adjacent job] + [adjacent job] in one tool"
- "[user type] + [platform A] + [platform B] + research"
- "[pain description] + solution/tool/app/software"

输出结果：研究对象卡片、四层关键词包（描述词汇 + 痛苦表达 + 替代行为 + 社区词汇）、平台优先级建议、平台词汇变换规则。

### 12.2 模块二：多平台采集

功能目标：跨平台聚合与归档相关信息。[cite:9][cite:10][cite:20]

核心功能：
- 自动搜索不同平台。
- 按平台回收结果摘要与原始链接。
- 识别竞品名称、产品页、评论页、讨论页。
- 支持时间过滤、地区过滤与语言过滤。
- 支持保存研究批次。

输出结果：原始证据池。

### 12.3 模块三：竞品识别与卡片化

功能目标：形成结构化竞品库，而非简单列表。[cite:9][cite:13]

竞品卡片字段：
- 名称
- 官网/商店链接
- 产品定位
- 目标用户
- 核心功能
- 定价模式
- 平台分布
- 评分与评论量
- 最近更新迹象
- 用户常见满意点
- 用户常见不满点
- 潜在切入机会

### 12.4 模块四：评论与讨论挖掘

功能目标：从评论、帖子、问答与讨论中提炼用户真实声音。[cite:17][cite:20][cite:23]

核心能力：
- 情绪判断：正向、中性、负向。
- 主题聚类：把分散文本聚合为常见主题。
- 高频抱怨识别：如价格高、体验差、学习成本高、缺功能、集成差、慢、不稳定。[cite:9][cite:17][cite:23]
- 建议提取：识别“希望增加”“如果能支持就好了”等表达。
- 替代方案识别：如用户仍用 Excel、Notion、邮件、微信群等原始方案。[cite:15]

### 12.5 模块五：信号抽取与统一标签体系

功能目标：把所有平台信息转成可比较的统一信号。

一级信号建议包括：
- 需求信号
- 痛点信号
- 替代信号
- 竞品信号
- 满意信号
- 不满信号
- 改进信号
- 付费信号
- 趋势信号
- 风险信号

每条信号需绑定：来源平台、原始链接、时间、主题标签、情绪、证据强度、代表性说明。[cite:20][cite:23]

### 12.6 模块六：价值评估模型

功能目标：输出标准化的“值不值得做”判断。[cite:13][cite:18][cite:21]

一级评估维度：

| 维度 | 解释 |
|---|---|
| 需求强度 | 用户是否频繁且明确表达该问题 [cite:15][cite:21] |
| 痛点强度 | 当前问题是否高痛、高频、高成本 [cite:15] |
| 市场拥挤度 | 竞品数量、成熟度、头部强度 [cite:9][cite:13] |
| 差异化空间 | 是否存在未满足的用户群、场景或功能空白 [cite:9][cite:13] |
| 用户不满密度 | 高质量负面反馈是否集中且重复 [cite:17][cite:23] |
| 趋势方向 | 需求与讨论是上升、平稳还是下降 [cite:10][cite:12] |
| 商业化可行性 | 是否已有付费行为与可接受价格带 [cite:17][cite:23] |
| 进入门槛 | 技术、品牌、监管、分发与网络效应门槛 [cite:12][cite:13] |
| MVP 可验证性 | 是否可低成本快速验证关键假设 [cite:18][cite:21] |

最终输出：总评估、关键机会、关键风险、不确定性说明。

### 12.7 模块七：策略生成

功能目标：根据不同市场局面给出不同做法，而不是统一模板。[cite:13][cite:18]

核心策略类型：
- 蓝海验证型：竞品少，需要先验证是否真有需求。[cite:18][cite:21]
- 差评切入型：竞品多但用户不满集中，适合针对痛点切入。[cite:9][cite:17][cite:23]
- 细分人群型：头部强，但细分用户未被照顾。[cite:13]
- 体验重构型：技术变化带来新的交互与成本结构。[cite:12]
- 暂缓进入型：市场过于成熟且切口不清晰。[cite:13]

### 12.8 模块八：验证行动生成

功能目标：将分析结果转化为立即可执行的验证方案。[cite:18][cite:21]

输出内容可包括：
- 访谈问题清单。
- Landing Page 核心文案。
- MVP 功能优先级建议。
- 广告测试文案。
- 种子用户招募建议。
- 手工服务验证方案。
- 一周或两周行动计划。[cite:18][cite:21]

## 13. 输出体系

系统输出分为四层。

### 13.1 结论层

- 是否建议做。
- 建议的进入方式。
- 最重要的三到五条依据。
- 最大风险与最大机会。[cite:13][cite:18][cite:21]

### 13.2 结构层

- 目标用户与场景。
- 需求地图。
- 替代方案地图。
- 竞品矩阵。
- 满意点/不满点主题。
- 趋势与风险概览。[cite:9][cite:10][cite:17]

### 13.3 策略层

- 定位建议。
- 差异化建议。
- 初版功能建议。
- 不建议做的功能。
- 对标竞品与切入口建议。[cite:13][cite:15]

### 13.4 行动层

- 访谈。
- 落地页。
- 原型。
- 广告验证。
- 种子用户测试。
- 验证节奏安排。[cite:18][cite:21]

## 14. 核心流程

### 14.1 主流程

1. 用户输入产品想法。
2. 系统完成想法建模，生成四层查询词汇包（描述词汇 + 痛苦表达 + 替代行为 + 社区词汇）。
3. **第一轮搜索（宽网捕词）**：系统按产品类型调用不同平台，使用原始建模词汇进行初步采集，此轮重点是从结果中提取社区原生词汇，而非追求精准匹配。
4. **词汇提取与精化**：系统从第一轮结果中提取高频共现词、与产品核心 job 相关的原生词汇、竞品被提及时的词汇语境，生成精化查询集。
5. **第二轮搜索（精准搜索）**：系统使用精化后的社区原生词汇重新构造查询，在各平台进行精准采集。
6. 系统合并两轮结果，去重后识别竞品与相关讨论。
7. 系统抽取需求、痛点、满意、不满、改进与风险等信号。
8. 系统形成竞品矩阵与主题聚类。
9. 系统进行多维评分与价值评估。
10. 系统生成结论、策略与验证计划。
11. 用户可保存项目、继续追加平台、修正假设或比较多个想法。

**注意**：对于有成熟类别名的产品（如 todo app），痛苦表达层和替代行为层的查询仍会生成但不作为主力，描述词汇层为主。对于无成熟类别名的产品，痛苦表达层和替代行为层是主力，描述词汇层为辅。系统需根据想法建模的类别置信度自动判断各层权重。

### 14.2 迭代流程

研究不应是一次性动作。用户应能在看到初步结果后补充约束、排除误判竞品、调整目标用户与场景，系统再重新采集与评分，以形成更贴近真实机会空间的判断。[cite:21]

## 16. 数据模型

### 16.1 核心对象

| 对象 | 说明 |
|---|---|
| ResearchProject | 一次完整调研项目 |
| IdeaModel | 标准化后的产品想法 |
| SourceResult | 某个平台采集到的原始结果 |
| Competitor | 识别出的竞品对象 |
| UserVoice | 一条评论、帖子、问答或反馈 |
| Signal | 从原始文本中抽取出的标准信号 |
| EvaluationScore | 各维度评分结果 |
| StrategySuggestion | 自动生成的策略建议 |
| ValidationPlan | 后续验证计划 |

### 16.2 关键关系

- 一个 ResearchProject 包含一个或多个 IdeaModel。
- 一个 IdeaModel 关联多个 SourceResult。
- 多个 SourceResult 可归并为同一个 Competitor。
- 一个 UserVoice 可对应多个 Signal。
- 多个 Signal 共同支撑一个 EvaluationScore。
- EvaluationScore 驱动 StrategySuggestion 与 ValidationPlan 的生成。

## 17. 评分与可信度机制

产品必须同时输出“结论”和“对结论的置信说明”。这部分是系统可信度的关键。

建议引入以下机制：
- 证据量：某一主题由多少条独立证据支撑。
- 跨平台一致性：该结论是否在不同平台重复出现。[cite:20][cite:23]
- 时间有效性：证据是近期还是陈旧。
- 来源质量：平台、内容类型与作者角色的权重不同。
- 样本偏差提示：如只来自单一社区或单一语言环境。[cite:20]

## 18. MVP 范围

第一版不应追求全覆盖，而应优先验证核心闭环：用户输入一个想法后，系统能不能稳定输出一份有价值的“是否值得做 + 为什么 + 怎么做”的研究结果。[cite:18][cite:21]

### 18.1 MVP 必做

- 想法建模。
- 多平台基础采集，优先搜索引擎、Reddit、应用商店、G2 类点评站。[cite:9][cite:17][cite:20]
- 竞品卡片生成。
- 评论/讨论主题聚类。
- 价值评估模型第一版。
- 研究报告页。
- 验证行动建议。

### 18.2 MVP 可延后

- 自动长期监控。
- 团队协作。
- 批量项目研究。
- 深度行业报告接入。
- 高级图表与自定义评分模型。
- 多语言自动翻译与跨市场对比。

## 19. 非目标范围

第一版不做以下内容：
- 全量行业数据库平台。
- 通用舆情监测系统。
- 完整 CRM 或销售情报系统。
- 广告投放执行平台。
- 纯自动化“替用户决定创业方向”的黑盒系统。

## 20. 风险与挑战

### 20.1 产品风险

- 不同平台数据结构差异大，难以稳定抽取。[cite:20]
- 评论和帖子噪声很大，容易误判主流问题。[cite:23]
- 用户容易期待“绝对正确答案”，但研究本质上仍有不确定性。[cite:21]
- 部分赛道信息公开程度低，系统证据可能不足。

### 20.2 解决思路

- 引入证据回链与原文查看。
- 为每个结论显示置信提示与证据量。
- 明确“结论 + 风险 + 待验证假设”的输出结构。[cite:18][cite:21]
- 采用分产品类型的平台权重模型。[cite:9][cite:17]

## 21. 成功指标

### 21.1 产品指标

- 从输入想法到生成首版结果的完成率。
- 用户查看完整报告的比例。
- 用户保存项目与二次编辑比例。
- 用户采纳验证计划的比例。
- 多项目比较功能使用率。

### 21.2 结果质量指标

- 用户对“竞品覆盖度”的主观满意度。
- 用户对“痛点提炼准确度”的主观满意度。
- 用户对“结论可执行性”的主观满意度。
- 研究结果被用于实际立项、调整定位或优化产品的比例。

## 22. 版本规划建议

### V1：研究 Copilot

核心目标是把零散调研流程串起来，完成想法建模、基础采集、竞品识别、信号提炼、初版评分与报告输出。[cite:9][cite:15][cite:21]

### V2：研究工作台

加入项目沉淀、竞品库、证据管理、多想法比较与更强的主题聚类能力，重点提升结构化研究效率。[cite:9][cite:17][cite:20]

### V3：决策系统

加入长期监控、竞品预警、自定义评分模型与更成熟的进入策略建议，重点从“研究工具”升级为“持续决策工具”。[cite:13][cite:18]

## 23. 最终产品定义

该产品的本质不是“替代搜索”，也不是“自动写一份市场分析”。它更接近一个围绕产品想法展开的研究操作系统：以前置研究为入口，以跨平台证据为原料，以统一信号体系为中间层，以价值评估和策略建议为最终输出，帮助用户在进入开发前更理性地判断做什么、不做什么、先验证什么。[cite:13][cite:15][cite:18][cite:21]
